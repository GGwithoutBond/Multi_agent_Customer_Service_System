# 1. 项目概述

## 1.1 项目背景

随着人工智能技术的快速发展,智能客服系统已成为企业提升服务效率、降低运营成本的重要工具。传统的单一Agent客服系统在处理复杂业务场景时存在局限性,难以应对多领域知识融合、长对话上下文维护等挑战。

本项目旨在构建一套基于多智能体协作的智能客服系统,采用Orchestrator-Worker架构模式,通过LangGraph实现智能体工作流编排,结合GraphRAG技术实现知识图谱增强的检索增强生成,为用户提供精准、高效、个性化的客服体验。

## 1.2 项目目标

-   构建可扩展的多智能体协作框架,支持灵活的Agent编排与管理

-   实现基于GraphRAG的知识检索系统,提升回答的准确性和相关性

-   设计高效的多轮对话记忆机制,保持对话上下文的连贯性

-   采用前后端分离架构,实现系统的高可用性和可维护性

-   支持多渠道接入(Web、App、微信等),统一客服服务入口

## 1.3 核心特性

  ----------------------- ------------------------------------------------------------------

  多智能体协作            Orchestrator统一调度,Worker专注特定领域,实现复杂任务的分解与协作

## 2.2 Orchestrator-Worker 架构详解

Orchestrator-Worker是一种主从协调的多智能体架构模式,其核心思想是将复杂任务分解为多个子任务,由专门的Worker
Agent并行或串行执行。

### 2.2.1 Orchestrator Agent(编排器)

-   **意图识别：**分析用户输入，识别用户意图和请求类型

-   **任务分解：**将复杂请求分解为可执行的子任务序列

-   **路由决策：**根据任务类型选择合适的Worker Agent

-   **结果聚合：**收集各Worker返回结果，合成最终响应

-   **状态管理：**维护对话状态和工作流执行状态

### 2.2.2 Worker Agents(工作智能体)

  ----------------------- ----------------------- -----------------------
  **Worker类型**          **职责**                **技术实现**

  FAQ Worker              处理常见问题查询        GraphRAG检索 + 模板匹配

  Order Worker            订单查询、物流追踪      数据库查询 + API调用

  Product Worker          产品咨询、推荐          GraphRAG + 向量检索

  Complaint Worker        投诉处理、工单创建      情感分析 + 工单系统

  Human Worker            转人工服务触发          会话转接 + 消息队列
  ----------------------- ----------------------- -----------------------

# 4. 核心模块设计

## 4.1 LangGraph 工作流设计

LangGraph是LangChain团队开发的图状态机框架，支持循环、分支、并行执行等复杂工作流模式。

### 4.1.1 状态定义

class AgentState(TypedDict):

messages: list\[BaseMessage\] \# 消息历史

intent: str \# 识别的意图

context: dict \# 上下文信息

memory: ConversationMemory \# 对话记忆

current_worker: str \# 当前Worker

response: str \# 最终响应

### 4.1.2 图结构定义

graph = StateGraph(AgentState)

\# 添加节点

graph.add_node(\'orchestrator\', orchestrator_node)

graph.add_node(\'faq_worker\', faq_worker_node)

graph.add_node(\'order_worker\', order_worker_node)

graph.add_node(\'response_generator\', response_node)

\# 条件路由

graph.add_conditional_edges(\'orchestrator\', route_to_worker)

graph.add_edge(\'faq_worker\', \'response_generator\')

graph.add_edge(\'order_worker\', \'response_generator\')

## 4.2 GraphRAG 知识检索设计

GraphRAG结合知识图谱与向量检索,通过实体关系增强检索的语义理解能力。

### 4.2.1 知识图谱构建流程

10. 文档解析:将知识库文档(FAQ、产品手册等)解析为结构化文本

11. 实体抽取:使用NER模型抽取实体(产品名、功能、参数等)

12. 关系构建:基于LLM进行关系三元组抽取(实体-关系-实体)

13. 图存储:将实体和关系存入Neo4j图数据库

14. 向量索引:对文本chunk进行向量化,存入Milvus

### 4.2.2 混合检索策略

  ----------------------- ----------------------- ------------------------
  **检索方式**            **实现方法**            **适用场景**

  向量检索                Milvus ANN搜索          语义相似度匹配

  图检索                  Neo4j Cypher查询        实体关系推理、多跳问答

  关键词检索              Elasticsearch BM25      精确匹配、术语检索

  混合排序                RRF融合算法             综合多路召回结果
  ----------------------- ----------------------- ------------------------

## 8.2 网络架构

所有服务运行在同一个Docker网络中,通过服务名进行内部通信。Nginx作为入口,处理外部流量并转发到对应服务。

互联网 ↓ Nginx (:80, :443) ├─→ Frontend (Vue应用) └─→ API Gateway
(:8000) ├─→ Chat Service (x3实例) │ ├─→ Agent Worker (x5实例) │ └─→ RAG
Service └─→ 数据层 ├─→ Redis (缓存) ├─→ RabbitMQ (消息队列) ├─→
PostgreSQL (主数据) ├─→ Neo4j (图数据) └─→ Milvus (向量数据)
